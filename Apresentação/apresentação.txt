# Slide 1 — What is a Doubly Linked List

> A Doubly Linked List is a data structure made of nodes connected to each other.
> Each node stores a value and two references: one to the next element and one to the previous element.
> This allows us to move through the structure both forward and backward.
> In Scala 3, we usually represent this using a `Node` class that stores the value and the `prev` and `next` references.

---

# Slide 2 — Why this structure was created

> The Doubly Linked List was created to solve some limitations of the singly linked list.
> In a singly linked list, we can only move through the elements in one direction.
> In the doubly linked version, each node knows both the next and the previous node.
> This makes operations like removal or reverse navigation easier.

---

# Slide 3 — When it is good to use

> This structure is useful when we need to move through the data in two directions.
> A common example is the navigation history of a web browser or undo and redo systems.
> Another case is when we need to insert or remove many elements in the middle of the structure.
> Because each node knows its neighbors, these operations can be done efficiently.

---

# Slide 4 — When it is not good to use

> Even with its advantages, this structure also has some disadvantages.
> Each node must store two pointers, which increases memory usage.
> Also, accessing an element by index is slow because we need to traverse the list.
> For this reason, when we need fast access by position, arrays or vectors are usually better.

---

# Slide 5 — Trade-offs

> Like any data structure, there is a trade-off.
> The Doubly Linked List provides a lot of flexibility for insertions, removals, and bidirectional navigation.
> However, this comes with higher memory usage and slower index access.
> Because of this, choosing the right structure depends on the type of operations the system needs.

---

# Slide 6 — Conclusion

> In summary, the Doubly Linked List is a dynamic and very flexible data structure.
> It is ideal when we need to frequently modify elements or move in both directions through the data.
> However, it is not the best choice when we need fast access by index.
> Understanding these trade-offs helps us choose the correct data structure for each problem.
